# HabitHamster — README по схеме базы данных

## Назначение схемы

Эта схема базы данных описывает продукт **HabitHamster** — приложение для отслеживания привычек, анализа прогресса и геймификации пользовательского поведения.

Основная идея схемы: пользователь создаёт привычки, отмечает их выполнение, а система на основе истории выполнений считает прогресс, streak, активность, достижения, челленджи и персональные инсайты.

## Главная логика продукта

```text
users
  ├── user_profiles
  ├── habits
  │     ├── habit_schedules
  │     ├── habit_logs
  │     └── habit_tags ─── tags ─── tag_categories
  │
  ├── user_achievements ─── achievements
  ├── user_challenges ─── challenges
  └── user_insights
```

Ключевая таблица для аналитики — `habit_logs`. Именно она хранит историю выполнения привычек. Из неё можно считать:

* процент выполнения привычек;
* streak;
* heatmap активности;
* общее время, потраченное на привычки;
* самые сильные и слабые дни недели;
* прогресс по тегам и категориям;
* выполнение челленджей;
* получение достижений.

# 1. `users`

## Для чего нужна таблица

Таблица `users` хранит пользователей приложения.

В Django эту таблицу можно не создавать вручную, потому что есть стандартная таблица пользователей `auth_user`. Но на уровне продуктовой схемы сущность `users` всё равно является центральной.

Пользователь может:

* создавать привычки;
* отмечать выполнение привычек;
* получать достижения;
* участвовать в челленджах;
* получать инсайты;
* иметь профиль с уровнем, XP и маскотом.

## Поля

| Поле            | Тип        | Назначение                                                |
| --------------- | ---------- | --------------------------------------------------------- |
| `id`            | bigint, PK | Уникальный идентификатор пользователя.                    |
| `email`         | varchar    | Email пользователя. Используется для входа и уведомлений. |
| `username`      | varchar    | Никнейм пользователя.                                     |
| `password_hash` | varchar    | Хеш пароля. Пароль в открытом виде не хранится.           |
| `is_active`     | boolean    | Активен ли аккаунт пользователя.                          |
| `created_at`    | timestamp  | Дата регистрации пользователя.                            |
| `updated_at`    | timestamp  | Дата последнего обновления данных пользователя.           |

## Основные связи

| Связь                         | Тип |
| ----------------------------- | --- |
| `users` → `user_profiles`     | 1:1 |
| `users` → `habits`            | 1:N |
| `users` → `habit_logs`        | 1:N |
| `users` → `user_achievements` | 1:N |
| `users` → `user_challenges`   | 1:N |
| `users` → `user_insights`     | 1:N |

# 2. `user_profiles`

## Для чего нужна таблица

Таблица `user_profiles` расширяет базовую информацию о пользователе.

Она нужна для хранения продуктовых данных, которые не относятся напрямую к авторизации:

* уровень пользователя;
* количество XP;
* текущий streak;
* лучший streak;
* данные маскота.

Эту таблицу лучше отделять от `users`, потому что `users` отвечает за аккаунт и авторизацию, а `user_profiles` — за игровые и продуктовые характеристики.

## Поля

| Поле             | Тип                     | Назначение                                                       |
| ---------------- | ----------------------- | ---------------------------------------------------------------- |
| `id`             | bigint, PK              | Уникальный идентификатор профиля.                                |
| `user_id`        | bigint, FK → `users.id` | Пользователь, которому принадлежит профиль.                      |
| `level`          | integer                 | Уровень пользователя. Растёт по мере накопления XP.              |
| `xp`             | integer                 | Количество опыта пользователя.                                   |
| `current_streak` | integer                 | Текущая серия успешных дней.                                     |
| `best_streak`    | integer                 | Лучшая серия успешных дней за всё время.                         |
| `mascot_name`    | varchar                 | Имя маскота пользователя.                                        |
| `mascot_mood`    | varchar                 | Текущее настроение маскота. Например: `happy`, `sad`, `neutral`. |
| `created_at`     | timestamp               | Дата создания профиля.                                           |
| `updated_at`     | timestamp               | Дата обновления профиля.                                         |

## Основные связи

| Связь                     | Тип |
| ------------------------- | --- |
| `users` → `user_profiles` | 1:1 |

## Пример

```text
user_id = 1
level = 3
xp = 240
current_streak = 5
best_streak = 12
mascot_name = "Хома"
mascot_mood = "happy"
```

# 3. `habits`

## Для чего нужна таблица

Таблица `habits` хранит привычки пользователя.

Это одна из главных таблиц продукта. Каждая запись описывает одну привычку: что пользователь хочет делать, в каком формате, с какой целью и как эта привычка должна отображаться в интерфейсе.

Примеры привычек:

* читать 30 минут в день;
* выпивать 8 стаканов воды;
* бегать 3 раза в неделю;
* делать зарядку каждый день;
* учить английский 20 минут.

## Поля

| Поле           | Тип                     | Назначение                                                              |
| -------------- | ----------------------- | ----------------------------------------------------------------------- |
| `id`           | bigint, PK              | Уникальный идентификатор привычки.                                      |
| `user_id`      | bigint, FK → `users.id` | Пользователь, которому принадлежит привычка.                            |
| `title`        | varchar                 | Название привычки. Например: `Читать книгу`.                            |
| `description`  | text                    | Описание привычки.                                                      |
| `target_type`  | varchar                 | Тип цели привычки. Например: `check`, `minutes`, `count`.               |
| `target_value` | integer                 | Целевое значение. Например: `30`, если цель — 30 минут.                 |
| `target_unit`  | varchar                 | Единица измерения. Например: `minutes`, `pages`, `steps`.               |
| `color`        | varchar                 | Цвет привычки в интерфейсе.                                             |
| `icon`         | varchar                 | Иконка привычки.                                                        |
| `is_active`    | boolean                 | Активна ли привычка. Если `false`, привычка скрыта из активного списка. |
| `created_at`   | timestamp               | Дата создания привычки.                                                 |
| `updated_at`   | timestamp               | Дата обновления привычки.                                               |

## Возможные значения `target_type`

| Значение  | Описание                                        |
| --------- | ----------------------------------------------- |
| `check`   | Простая отметка выполнения: сделал / не сделал. |
| `minutes` | Привычка измеряется во времени.                 |
| `count`   | Привычка измеряется количеством повторений.     |

## Возможные значения `target_unit`

| Значение     | Описание        |
| ------------ | --------------- |
| `times`      | Количество раз. |
| `minutes`    | Минуты.         |
| `pages`      | Страницы.       |
| `steps`      | Шаги.           |
| `glasses`    | Стаканы.        |
| `kilometers` | Километры.      |

## Основные связи

| Связь                        | Тип                    |
| ---------------------------- | ---------------------- |
| `users` → `habits`           | 1:N                    |
| `habits` → `habit_schedules` | 1:1                    |
| `habits` → `habit_logs`      | 1:N                    |
| `habits` → `habit_tags`      | 1:N                    |
| `habits` ↔ `tags`            | M:N через `habit_tags` |

## Пример

```text
title = "Читать книгу"
target_type = "minutes"
target_value = 30
target_unit = "minutes"
```

# 4. `habit_schedules`

## Для чего нужна таблица

Таблица `habit_schedules` хранит расписание привычки.

Привычка и её расписание разделены, потому что привычка описывает **что нужно делать**, а расписание описывает **когда и как часто это нужно делать**.

Это позволяет поддерживать разные варианты:

* каждый день;
* только по будням;
* несколько раз в неделю;
* раз в месяц;
* кастомное расписание.

## Поля

| Поле               | Тип                      | Назначение                                                 |
| ------------------ | ------------------------ | ---------------------------------------------------------- |
| `id`               | bigint, PK               | Уникальный идентификатор расписания.                       |
| `habit_id`         | bigint, FK → `habits.id` | Привычка, к которой относится расписание.                  |
| `frequency_type`   | varchar                  | Тип частоты: `daily`, `weekly`, `monthly`, `custom`.       |
| `days_of_week`     | json / array             | Дни недели, в которые активна привычка.                    |
| `times_per_period` | integer                  | Сколько раз нужно выполнить привычку за период.            |
| `reminder_time`    | time, nullable           | Время напоминания.                                         |
| `start_date`       | date                     | Дата начала привычки.                                      |
| `end_date`         | date, nullable           | Дата окончания привычки. Если пусто — привычка бессрочная. |

## Возможные значения `frequency_type`

| Значение  | Описание                     |
| --------- | ---------------------------- |
| `daily`   | Каждый день.                 |
| `weekly`  | По определённым дням недели. |
| `monthly` | По определённым дням месяца. |
| `custom`  | Кастомная логика расписания. |

## Основные связи

| Связь                        | Тип |
| ---------------------------- | --- |
| `habits` → `habit_schedules` | 1:1 |

## Пример ежедневной привычки

```text
habit_id = 1
frequency_type = "daily"
days_of_week = [1, 2, 3, 4, 5, 6, 7]
times_per_period = 1
```

## Пример привычки по будням

```text
habit_id = 2
frequency_type = "weekly"
days_of_week = [1, 2, 3, 4, 5]
times_per_period = 5
```

# 5. `habit_logs`

## Для чего нужна таблица

Таблица `habit_logs` хранит историю выполнения привычек.

Это самая важная таблица для аналитики. Каждая запись показывает, что произошло с конкретной привычкой в конкретный день.

На основе этой таблицы можно считать:

* выполнена ли привычка сегодня;
* процент выполнения за неделю или месяц;
* streak;
* heatmap активности;
* общее количество выполнений;
* суммарное время по привычкам;
* прогресс по тегам;
* выполнение достижений и челленджей.

## Поля

| Поле               | Тип                      | Назначение                                                          |
| ------------------ | ------------------------ | ------------------------------------------------------------------- |
| `id`               | bigint, PK               | Уникальный идентификатор записи.                                    |
| `habit_id`         | bigint, FK → `habits.id` | Привычка, к которой относится запись.                               |
| `user_id`          | bigint, FK → `users.id`  | Пользователь, которому принадлежит запись.                          |
| `log_date`         | date                     | Дата выполнения привычки.                                           |
| `status`           | varchar                  | Статус выполнения: `done`, `missed`, `skipped`, `partial`.          |
| `value`            | integer                  | Фактическое значение выполнения. Например: 20 минут или 5 стаканов. |
| `duration_minutes` | integer                  | Сколько минут было потрачено.                                       |
| `note`             | text, nullable           | Заметка пользователя.                                               |
| `mood_score`       | integer, nullable        | Оценка настроения от 1 до 5.                                        |
| `created_at`       | timestamp                | Дата создания записи.                                               |
| `updated_at`       | timestamp                | Дата обновления записи.                                             |

## Возможные значения `status`

| Значение  | Описание                                                          |
| --------- | ----------------------------------------------------------------- |
| `done`    | Привычка выполнена.                                               |
| `missed`  | Привычка пропущена.                                               |
| `skipped` | Привычка пропущена осознанно, например из-за болезни или отпуска. |
| `partial` | Привычка выполнена частично.                                      |

## Основные связи

| Связь                   | Тип |
| ----------------------- | --- |
| `habits` → `habit_logs` | 1:N |
| `users` → `habit_logs`  | 1:N |

## Ограничения

```text
UNIQUE(habit_id, log_date)
```

Это значит, что у одной привычки может быть только одна запись за один день.

## Пример

```text
habit_id = 1
user_id = 1
log_date = 2026-05-07
status = "done"
value = 30
duration_minutes = 30
```

# 6. `tag_categories`

## Для чего нужна таблица

Таблица `tag_categories` хранит категории тегов.

Категории нужны, чтобы группировать привычки для аналитики. Например, система сможет понять, что пользователь лучше выполняет привычки из категории `здоровье`, но хуже — из категории `обучение`.

Категории могут быть вложенными через `parent_id`.

Пример структуры:

```text
Здоровье
  └── Физическая активность
      └── Кардио

Обучение
  └── Чтение
```

## Поля

| Поле         | Тип                                        | Назначение                                            |
| ------------ | ------------------------------------------ | ----------------------------------------------------- |
| `id`         | bigint, PK                                 | Уникальный идентификатор категории.                   |
| `parent_id`  | bigint, FK → `tag_categories.id`, nullable | Родительская категория. Используется для вложенности. |
| `name`       | varchar                                    | Название категории.                                   |
| `slug`       | varchar                                    | Системное имя категории. Например: `health`, `sport`. |
| `created_at` | timestamp                                  | Дата создания категории.                              |

## Основные связи

| Связь                               | Тип                    |
| ----------------------------------- | ---------------------- |
| `tag_categories` → `tag_categories` | 1:N, parent / children |
| `tag_categories` → `tags`           | 1:N                    |

# 7. `tags`

## Для чего нужна таблица

Таблица `tags` хранит конкретные теги привычек.

Теги нужны для:

* фильтрации привычек;
* группировки привычек;
* аналитики по направлениям;
* челленджей по категориям;
* персональных инсайтов.

Примеры тегов:

* `кардио`;
* `чтение`;
* `сон`;
* `питание`;
* `учёба`;
* `спорт`.

## Поля

| Поле          | Тип                              | Назначение                          |
| ------------- | -------------------------------- | ----------------------------------- |
| `id`          | bigint, PK                       | Уникальный идентификатор тега.      |
| `category_id` | bigint, FK → `tag_categories.id` | Категория, к которой относится тег. |
| `name`        | varchar                          | Название тега.                      |
| `slug`        | varchar                          | Системное имя тега.                 |
| `created_at`  | timestamp                        | Дата создания тега.                 |

## Основные связи

| Связь                     | Тип                    |
| ------------------------- | ---------------------- |
| `tag_categories` → `tags` | 1:N                    |
| `tags` → `habit_tags`     | 1:N                    |
| `tags` ↔ `habits`         | M:N через `habit_tags` |

# 8. `habit_tags`

## Для чего нужна таблица

Таблица `habit_tags` связывает привычки и теги.

Так как одна привычка может иметь несколько тегов, и один тег может использоваться во многих привычках, между `habits` и `tags` возникает связь many-to-many.

Для такой связи нужна промежуточная таблица `habit_tags`.

## Поля

| Поле       | Тип                      | Назначение                      |
| ---------- | ------------------------ | ------------------------------- |
| `id`       | bigint, PK               | Уникальный идентификатор связи. |
| `habit_id` | bigint, FK → `habits.id` | Привычка.                       |
| `tag_id`   | bigint, FK → `tags.id`   | Тег.                            |

## Основные связи

| Связь                   | Тип                    |
| ----------------------- | ---------------------- |
| `habits` → `habit_tags` | 1:N                    |
| `tags` → `habit_tags`   | 1:N                    |
| `habits` ↔ `tags`       | M:N через `habit_tags` |

## Ограничения

```text
UNIQUE(habit_id, tag_id)
```

Это значит, что один и тот же тег нельзя привязать к одной привычке дважды.

## Пример

```text
habit = "Бегать"
tags = ["спорт", "здоровье", "кардио"]
```

# 9. `achievements`

## Для чего нужна таблица

Таблица `achievements` хранит справочник достижений.

Это не достижения конкретного пользователя, а список всех возможных достижений в системе.

Примеры достижений:

* `7 дней подряд`;
* `100 выполнений`;
* `10 часов активности`;
* `Первый челлендж завершён`.

## Поля

| Поле              | Тип               | Назначение                                             |
| ----------------- | ----------------- | ------------------------------------------------------ |
| `id`              | bigint, PK        | Уникальный идентификатор достижения.                   |
| `code`            | varchar           | Системный код достижения. Например: `streak_7`.        |
| `title`           | varchar           | Название достижения.                                   |
| `description`     | text              | Описание достижения.                                   |
| `condition_type`  | varchar           | Тип условия получения.                                 |
| `condition_value` | integer           | Значение условия. Например: 7 дней или 100 выполнений. |
| `xp_reward`       | integer           | Количество XP, которое получает пользователь.          |
| `icon`            | varchar, nullable | Иконка достижения.                                     |
| `created_at`      | timestamp         | Дата создания достижения.                              |

## Возможные значения `condition_type`

| Значение           | Описание                                  |
| ------------------ | ----------------------------------------- |
| `streak`           | Достижение за серию дней.                 |
| `completion_count` | Достижение за количество выполнений.      |
| `total_time`       | Достижение за суммарное время активности. |
| `xp`               | Достижение за количество XP.              |
| `custom`           | Кастомная логика получения.               |

## Основные связи

| Связь                                | Тип |
| ------------------------------------ | --- |
| `achievements` → `user_achievements` | 1:N |

## Пример

```text
code = "streak_7"
title = "7 дней подряд"
condition_type = "streak"
condition_value = 7
xp_reward = 100
```

# 10. `user_achievements`

## Для чего нужна таблица

Таблица `user_achievements` хранит достижения, которые уже получил конкретный пользователь.

Это промежуточная таблица между `users` и `achievements`.

Важно: сама таблица `achievements` хранит справочник достижений, а `user_achievements` хранит факт получения достижения пользователем.

## Поля

| Поле             | Тип                            | Назначение                                |
| ---------------- | ------------------------------ | ----------------------------------------- |
| `id`             | bigint, PK                     | Уникальный идентификатор записи.          |
| `user_id`        | bigint, FK → `users.id`        | Пользователь, который получил достижение. |
| `achievement_id` | bigint, FK → `achievements.id` | Полученное достижение.                    |
| `unlocked_at`    | timestamp                      | Дата и время получения достижения.        |

## Основные связи

| Связь                                | Тип                           |
| ------------------------------------ | ----------------------------- |
| `users` → `user_achievements`        | 1:N                           |
| `achievements` → `user_achievements` | 1:N                           |
| `users` ↔ `achievements`             | M:N через `user_achievements` |

## Ограничения

```text
UNIQUE(user_id, achievement_id)
```

Это значит, что один пользователь не может получить одно и то же достижение дважды.

## Пример

```text
user_id = 15
achievement_id = 1
unlocked_at = 2026-05-07 12:30:00
```

# 11. `challenges`

## Для чего нужна таблица

Таблица `challenges` хранит челленджи.

Челлендж — это ограниченная по времени или условию задача, которую пользователь может выполнить ради награды.

Примеры челленджей:

* `5 тренировок за неделю`;
* `7 дней без пропусков`;
* `10 часов чтения за месяц`;
* `3 учебные привычки за неделю`.

## Поля

| Поле              | Тип            | Назначение                          |
| ----------------- | -------------- | ----------------------------------- |
| `id`              | bigint, PK     | Уникальный идентификатор челленджа. |
| `title`           | varchar        | Название челленджа.                 |
| `description`     | text           | Описание челленджа.                 |
| `condition_type`  | varchar        | Тип условия выполнения.             |
| `condition_value` | integer        | Значение условия.                   |
| `xp_reward`       | integer        | Награда XP за выполнение.           |
| `start_date`      | date, nullable | Дата начала челленджа.              |
| `end_date`        | date, nullable | Дата окончания челленджа.           |
| `is_active`       | boolean        | Активен ли челлендж.                |
| `created_at`      | timestamp      | Дата создания челленджа.            |

## Возможные значения `condition_type`

| Значение               | Описание                                                       |
| ---------------------- | -------------------------------------------------------------- |
| `completion_count`     | Выполнить определённое количество привычек.                    |
| `tag_completion_count` | Выполнить определённое количество привычек с конкретным тегом. |
| `streak`               | Держать streak заданное количество дней.                       |
| `total_time`           | Набрать определённое количество минут активности.              |
| `custom`               | Кастомная логика выполнения.                                   |

## Основные связи

| Связь                            | Тип |
| -------------------------------- | --- |
| `challenges` → `user_challenges` | 1:N |

## Пример

```text
title = "5 тренировок за неделю"
condition_type = "tag_completion_count"
condition_value = 5
xp_reward = 200
```

# 12. `user_challenges`

## Для чего нужна таблица

Таблица `user_challenges` хранит участие пользователей в челленджах.

Она показывает:

* кто участвует в челлендже;
* какой у пользователя прогресс;
* завершён челлендж или нет;
* когда пользователь присоединился;
* когда он завершил челлендж.

## Поля

| Поле           | Тип                          | Назначение                                   |
| -------------- | ---------------------------- | -------------------------------------------- |
| `id`           | bigint, PK                   | Уникальный идентификатор записи.             |
| `user_id`      | bigint, FK → `users.id`      | Пользователь, который участвует в челлендже. |
| `challenge_id` | bigint, FK → `challenges.id` | Челлендж.                                    |
| `status`       | varchar                      | Статус участия.                              |
| `progress`     | integer                      | Текущий прогресс пользователя.               |
| `joined_at`    | timestamp                    | Дата присоединения к челленджу.              |
| `completed_at` | timestamp, nullable          | Дата завершения челленджа.                   |

## Возможные значения `status`

| Значение    | Описание                               |
| ----------- | -------------------------------------- |
| `active`    | Пользователь сейчас проходит челлендж. |
| `completed` | Пользователь выполнил челлендж.        |
| `failed`    | Пользователь провалил челлендж.        |
| `cancelled` | Пользователь отменил участие.          |

## Основные связи

| Связь                            | Тип                         |
| -------------------------------- | --------------------------- |
| `users` → `user_challenges`      | 1:N                         |
| `challenges` → `user_challenges` | 1:N                         |
| `users` ↔ `challenges`           | M:N через `user_challenges` |

## Ограничения

```text
UNIQUE(user_id, challenge_id)
```

Это значит, что один пользователь не может участвовать в одном и том же челлендже дважды одновременно.

# 13. `user_insights`

## Для чего нужна таблица

Таблица `user_insights` хранит персональные сообщения аналитики для пользователя.

Инсайты могут отображаться в интерфейсе или использоваться маскотом для обратной связи.

Примеры инсайтов:

* `Ты чаще выполняешь привычки утром`;
* `Воскресенье — самый слабый день`;
* `Ты на серии 5 дней подряд`;
* `Спортивные привычки выполняются лучше учебных`;
* `Маскот рад твоему прогрессу`.

## Поля

| Поле           | Тип                     | Назначение                                     |
| -------------- | ----------------------- | ---------------------------------------------- |
| `id`           | bigint, PK              | Уникальный идентификатор инсайта.              |
| `user_id`      | bigint, FK → `users.id` | Пользователь, которому принадлежит инсайт.     |
| `insight_type` | varchar                 | Тип инсайта.                                   |
| `title`        | varchar                 | Заголовок инсайта.                             |
| `message`      | text                    | Текст сообщения.                               |
| `payload`      | json                    | Дополнительные данные для логики или графиков. |
| `period_start` | date, nullable          | Начало анализируемого периода.                 |
| `period_end`   | date, nullable          | Конец анализируемого периода.                  |
| `is_read`      | boolean                 | Прочитал ли пользователь инсайт.               |
| `created_at`   | timestamp               | Дата создания инсайта.                         |

## Возможные значения `insight_type`

| Значение         | Описание                           |
| ---------------- | ---------------------------------- |
| `weak_day`       | Самый слабый день недели.          |
| `best_day`       | Самый успешный день недели.        |
| `streak`         | Инсайт про серию выполнений.       |
| `tag_analytics`  | Аналитика по тегам или категориям. |
| `mascot_message` | Сообщение от маскота.              |
| `custom`         | Другое сообщение.                  |

## Основные связи

| Связь                     | Тип |
| ------------------------- | --- |
| `users` → `user_insights` | 1:N |

## Пример

```text
insight_type = "weak_day"
title = "Слабый день недели"
message = "По воскресеньям ты выполняешь привычки реже всего"
payload = {"weekday": "sunday", "completion_rate": 0.34}
```

# Основные связи всей схемы

| Левая таблица    | Правая таблица      | Тип связи | Описание                                                    |
| ---------------- | ------------------- | --------- | ----------------------------------------------------------- |
| `users`          | `user_profiles`     | 1:1       | Один пользователь имеет один профиль.                       |
| `users`          | `habits`            | 1:N       | Один пользователь может создать много привычек.             |
| `habits`         | `habit_schedules`   | 1:1       | У одной привычки одно расписание.                           |
| `habits`         | `habit_logs`        | 1:N       | У одной привычки много записей выполнения.                  |
| `users`          | `habit_logs`        | 1:N       | Один пользователь имеет много записей выполнения.           |
| `tag_categories` | `tag_categories`    | 1:N       | Категория может иметь дочерние категории.                   |
| `tag_categories` | `tags`              | 1:N       | В одной категории может быть много тегов.                   |

| `habits`         | `tags`              | M:N       | Через таблицу `habit_tags`.                                 |
| `users`          | `user_achievements` | 1:N       | Один пользователь может получить много достижений.          |
| `achievements`   | `user_achievements` | 1:N       | Одно достижение может быть получено многими пользователями. |
| `users`          | `achievements`      | M:N       | Через таблицу `user_achievements`.                          |
| `users`          | `user_challenges`   | 1:N       | Один пользователь может участвовать во многих челленджах.   |
| `challenges`     | `user_challenges`   | 1:N       | Один челлендж может иметь много участников.                 |
| `users`          | `challenges`        | M:N       | Через таблицу `user_challenges`.                            |
| `users`          | `user_insights`     | 1:N       | Один пользователь может иметь много инсайтов.               |

# Основные ограничения

## `habit_logs`

```text
UNIQUE(habit_id, log_date)
```

Одна привычка не может иметь две записи выполнения за один день.

## `habit_tags`

```text
UNIQUE(habit_id, tag_id)
```

Один тег нельзя привязать к одной привычке дважды.

## `user_achievements`

```text
UNIQUE(user_id, achievement_id)
```

Один пользователь не может получить одно и то же достижение дважды.

## `user_challenges`

```text
UNIQUE(user_id, challenge_id)
```

Один пользователь не может дважды участвовать в одном и том же челлендже.

# Что считать из этой схемы

## Процент выполнения привычки

Берём `habit_logs` за период и считаем отношение выполненных дней к ожидаемым дням.

```text
completion_percent = done_logs / expected_logs * 100
```

## Streak

Берём последовательные записи `habit_logs` со статусом `done` по датам.

```text
streak = количество последовательных дней со status = "done"
```

## Heatmap

Группируем `habit_logs` по `log_date`.

```text
log_date → количество выполнений за день
```

## Общее время активности

Суммируем `duration_minutes`.

```text
total_time = SUM(habit_logs.duration_minutes)
```

## Аналитика по тегам

Связываем таблицы:

```text
habit_logs → habits → habit_tags → tags → tag_categories
```

Так можно понять, какие категории привычек пользователь выполняет лучше.

# Итог

Эта схема покрывает первую полноценную продуктовую версию HabitHamster:

* пользователи;
* профили;
* привычки;
* расписания;
* история выполнения;
* теги и категории;
* достижения;
* челленджи;
* инсайты;
* XP, уровни и маскот.

Главное преимущество схемы — вся аналитика строится на `habit_logs`, а не хранится в отдельной статичной таблице. Это делает систему гибкой и расширяемой.
